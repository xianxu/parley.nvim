Here's a clean starting point that stays simple but handles the six concerns you raised:

### Core Insight: Separate Identity from Assignment

The key modeling decision is treating **people as stable entities** and **team membership as time-bound assignments**. This naturally handles reorgs — you don't delete/recreate a person, you close one assignment and open another.

### Schema: Team File

```yaml
# teams/sync-engine.yaml
team: sync-engine

members:
  - id: alice
    role: tl        # tl | ic | manager
    level: L6
    skills: [elixir, postgres, ehr-integrations]
    joined: 2025-Q1
    left: null       # null = still here. set to 2026-Q2 when reorg happens
    absences:
      - type: parental-leave
        from: 2026-06-01
        to: 2026-09-01

  - id: bob
    role: ic
    level: L4
    skills: [elixir, ruby, api-design]
    joined: 2025-Q3
    left: null
    absences: []

projects:
  - id: fhir-v2-migration
    size: XL
    skills_needed: [elixir, ehr-integrations]
    tl_demand: 0.5    # fraction of a TL-quarter
    ic_demand: 2.0    # engineer-quarters needed this quarter

  - id: sync-oncall
    size: M
    recurring: quarterly   # appears every quarter automatically
    skills_needed: [elixir, postgres]
    tl_demand: 0.1
    ic_demand: 0.5
```

### Why This Works for Your Six Concerns

| Concern | How it's handled |
|---|---|
| **Reorgs** | Set `left: 2026-Q2` on old team file, add new entry in new team file. Person `id` is the join key — history is preserved across files |
| **Mid-quarter moves** | `joined`/`left` accept either quarter (`2026-Q2`) or exact date (`2026-04-15`). Your rollup logic can prorate |
| **Time off** | `absences` array. Future entries are just planned capacity reductions |
| **Skill matching** | `skills` on people, `skills_needed` on projects. Validation can flag gaps |
| **TL bandwidth** | `tl_demand` per project. A L6 TL might have capacity `1.0` per quarter, a L7 `1.5`. You compute headroom as `capacity - sum(tl_demand)` |
| **KTLO / recurring** | `recurring: quarterly` flag. Your tooling duplicates these into each quarter's plan automatically. Same cost model as any project |

### The One Derived View That Matters

From these files you can compute **the capacity gap per team per quarter**:

```
available = sum(member.capacity for active, non-absent members)
committed = sum(project.tl_demand + project.ic_demand)
gap       = available - committed
```

Where `member.capacity` is a function of role + level + absences:

```yaml
# config/capacity.yaml
capacity_defaults:
  tl:
    L6: { tl_budget: 1.0, ic_budget: 0.2 }   # TLs do some IC work too
    L7: { tl_budget: 1.5, ic_budget: 0.0 }
  ic:
    L4: { tl_budget: 0.0, ic_budget: 1.0 }
    L5: { tl_budget: 0.2, ic_budget: 0.8 }   # senior ICs lead small things
```

This gives you a simple **supply/demand** model: projects are demand, people are supply, skills are the constraint, and time is the axis.

### What I'd Resist Adding (For Now)

- **Dependency between people** (mentorship, ramp-up time) — real but adds combinatorial complexity
- **Percentage allocation per project per person** — tempting but becomes a spreadsheet you'll never maintain; demand at the team level is enough
- **Velocity/productivity multipliers** — politically toxic and false precision

Start here. The moment you can render "Team X is 0.3 engineer-quarters short in Q3, and has no one with `ehr-integrations` skill" — that's already more than most orgs have, and it's the exact artifact that bridges the business/tech conversation.
