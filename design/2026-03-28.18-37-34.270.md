### What OpenShell Is

OpenShell sits between your agent and your infrastructure. It governs how the agent executes, what the agent can see and do, and where inference goes. It enables agents to run in isolated sandboxes, giving you fine-grained control over privacy and security.

The key architectural insight: the core decision is **out-of-process policy enforcement**. Instead of relying on behavioral prompts, it enforces constraints on the environment the agent runs in — meaning the agent cannot override them, even if compromised. This is the "browser tab model" applied to agents.

Tools like Claude Code and Cursor ship with valuable internal guardrails and system prompts, but those protections live inside the agent. OpenShell wraps those harnesses, moving the ultimate control point entirely outside the agent's reach.

### Your Workflow Model — Mostly Right

Yes, your understanding is basically correct. You run one command like `openshell sandbox create -- claude`, and then any agent — Claude Code, Codex, etc. — runs unmodified inside OpenShell. The agent clones repos, writes code, runs tests *inside* the sandbox. You get:

- **Declarative YAML policies.** Static sections (filesystem, process) are locked at creation; dynamic sections (network, inference) can be hot-reloaded on a running sandbox with `openshell policy set`.
- **Network control** — you can block external inference hosts (api.openai.com, api.anthropic.com, etc.) and any accidental direct call to a cloud provider is blocked at the proxy.
- Every outbound connection is intercepted: **allowed** if it matches policy, **routed** for inference (strips caller creds, injects backend creds), or **denied** and logged.

So yes — the sandbox is your "RL gym." The agent can break things inside without touching your host.

### File Access — This Is Where It Gets Nuanced

OpenShell does **not** expose a simple Docker bind-mount from host → sandbox. The isolation is intentionally strong. Here's what you *can* do:

**1. Upload/Download (explicit file transfer)**

The docs show dedicated commands for file transfer:

```bash
# Upload files from host into sandbox
openshell sandbox upload my-sandbox ./src /sandbox/src

# Download files from sandbox to host
openshell sandbox download my-sandbox /sandbox/output ./local
```

You can also upload files at creation time with the `--upload` flag on `openshell sandbox create`.

**2. `--sync` flag (git-aware sync)**

The `--sync` flag uploads all git-tracked and untracked (non-ignored) files to `/sandbox` before the command runs. Under the hood, this invokes `sandbox_sync_up()` which uses `git ls-files` to collect paths and streams them via tar-over-SSH.

This is probably closest to what you want for the "map my local checkout" use case — but it's a **copy**, not a live mount.

**3. Editor integration (VS Code / Cursor over SSH)**

When `--editor` is used, OpenShell keeps the sandbox alive and installs an OpenShell-managed SSH include file. So you can do:

```bash
openshell sandbox create --editor vscode --name my-sandbox
openshell sandbox connect my-sandbox --editor cursor
```

This gives you VS Code Remote-SSH into the sandbox — so you edit files *inside the sandbox* using your normal Mac tools, but the files live in the sandbox filesystem. This is probably the most ergonomic path for you.

**4. SSH config export**

You can also generate raw SSH config for any tool:
```bash
openshell sandbox ssh-config my-sandbox
```

**5. OpenClaw's mirror/remote modes (if using OpenClaw)**

If you use OpenClaw, there's a "mirror" mode: before exec, it syncs the local workspace into the sandbox; after exec, syncs back. File tools operate through the sandbox bridge, but the local workspace remains the source of truth between turns. There's also a "remote" mode where the sandbox workspace becomes canonical after initial seed.

### Why No Direct Bind Mount

This is by design. Landlock LSM keeps the agent confined to declared paths, preventing reads of sensitive host files. A live bind mount would undermine the filesystem isolation that's one of the four policy domains. The sandbox is designed specifically for long-running, self-evolving agents — it is not generic container isolation.

### Your "Unified Proxy" Idea

You're thinking along exactly the right lines. OpenShell already has a primitive for this: the Policy Engine evaluates every agent action at the binary, destination, method, and path level; and the Privacy Router determines whether context stays on-device or routes to frontier models, based on organizational policy rather than the agent's judgment. The Privacy Router draws on NVIDIA's acquisition of Gretel, whose differential privacy technology strips PII from prompts before they reach external APIs.

So your semantic API filtering proxy could either layer on top of OpenShell's existing network policy + inference routing, or extend it.

### Practical Recommendation

For your workflow on Mac:

1. `openshell sandbox create --editor vscode --sync -- claude` — sync your git checkout in, use VS Code Remote-SSH for editing
2. Define your `policy.yaml` to control network egress precisely
3. Use `openshell sandbox upload/download` for ad-hoc file moves
4. Accept that the file boundary is intentionally an air gap, not a mount — this *is* the security model

### Maturity Caveat

This is alpha software in single-player mode — "proof-of-life: one developer, one environment, one gateway." They're building toward multi-tenant enterprise deployments, but expect rough edges.

### What `--editor` actually does

The `--editor` flag opens the sandbox in VS Code or Cursor — it generates a temporary SSH config entry and invokes the editor's Remote-SSH mechanism. When `--editor` is used, OpenShell keeps the sandbox alive and installs an OpenShell-managed SSH include file instead of cluttering your main `~/.ssh/config`.

That's it. It's a thin shim. For nvim, you can do the same thing manually in ~10 seconds.

### What to do instead

**Option 1: SSH in and run nvim inside the sandbox** (simplest)

```bash
# Create sandbox, keep it alive
openshell sandbox create --name my-sandbox --sync

# Get SSH config
openshell sandbox ssh-config my-sandbox >> ~/.ssh/config
# or just connect directly:
openshell sandbox connect my-sandbox

# Inside the sandbox, run nvim
nvim .
```

The sandbox base image is a full Linux environment. You can install nvim inside it, or better — build a custom sandbox image (`--from ./my-sandbox-dir`) with your nvim config baked in.

**Option 2: SSHFS mount + local nvim** (edit locally, files live in sandbox)

```bash
# Get the SSH host name
openshell sandbox ssh-config my-sandbox
# Mount the sandbox filesystem locally
mkdir -p ~/mnt/sandbox
sshfs openshell-my-sandbox:/sandbox ~/mnt/sandbox

# Now use local nvim
nvim ~/mnt/sandbox/
```

**Option 3: [remote-sshfs.nvim](https://github.com/nosduco/remote-sshfs.nvim)** (nvim-native)

remote-sshfs.nvim allows you to edit files on remote hosts via SSHFS as if they were local, loosely based on the VSCode Remote-SSH extension. Point it at the OpenShell SSH host and you get telescope integration, live grep, etc — all running over SSH to the sandbox.

**Option 4: tmux inside sandbox** (most ergonomic for nvim users)

```bash
openshell sandbox connect my-sandbox
# Inside:
tmux new -s dev
nvim .
```

This is probably the most natural fit. You get session persistence, and if you disconnect, your nvim session survives inside tmux in the sandbox.

### Custom sandbox image with your nvim config

Create a directory like `./my-sandbox/` with a `Dockerfile` and `policy.yaml`:

```dockerfile
FROM ghcr.io/nvidia/openshell-community/base:latest

RUN apt-get update && apt-get install -y neovim tmux ripgrep fd-find

# Copy your nvim config
COPY nvim/ /home/sandbox/.config/nvim/
```

Then:
```bash
openshell sandbox create --from ./my-sandbox --sync -- bash
```

This way every sandbox you spin up has your nvim config ready to go.

### Dotfiles: Mac vs Linux isn't that bad

Most of what matters for an agentic coding sandbox is cross-platform already:

- **nvim config** — pure Lua/vimscript, works identically on Linux
- **tmux.conf** — identical
- **shell aliases / .zshrc / .bashrc** — 95% portable
- **ripgrep, fd, fzf configs** — identical

The only Mac-specific bits are usually GUI tools (Hammerspoon, Raycast, etc.) and `brew`-isms — none of which matter inside a sandbox. You likely just need to copy your `~/.config/nvim/` and `~/.tmux.conf` into the image and it works.

### Repo structure

```
my-project/
├── .openshell/
│   ├── Dockerfile
│   ├── policy.yaml
│   └── dotfiles/
│       ├── nvim/          # symlink or copy of your ~/.config/nvim
│       ├── tmux.conf
│       └── bashrc
├── bootstrap.sh           # the "one command"
├── src/
└── ...
```

### Dockerfile

```dockerfile
FROM ghcr.io/nvidia/openshell-community/base:latest

# System deps
RUN apt-get update && apt-get install -y \
    neovim \
    tmux \
    ripgrep \
    fd-find \
    git \
    curl \
    unzip \
    build-essential \
    python3 python3-pip \
    nodejs npm \
    && rm -rf /var/lib/apt/lists/*

# Dotfiles
COPY dotfiles/nvim/ /home/sandbox/.config/nvim/
COPY dotfiles/tmux.conf /home/sandbox/.tmux.conf
COPY dotfiles/bashrc /home/sandbox/.bashrc

# Fix ownership
RUN chown -R sandbox:sandbox /home/sandbox/
```

### policy.yaml

```yaml
version: "1.0"

filesystem:
  writable:
    - /sandbox
    - /tmp
  readable:
    - /usr
    - /etc
    - /home/sandbox

network:
  egress:
    allow:
      - "github.com:443"
      - "registry.npmjs.org:443"
      - "pypi.org:443"
      # add your package registries
    deny:
      - "*"  # deny everything else by default
```

### bootstrap.sh

```bash
#!/usr/bin/env bash
set -euo pipefail

SANDBOX_NAME="${1:-$(basename $(pwd))}"

echo "Building sandbox image..."
openshell sandbox create \
    --name "$SANDBOX_NAME" \
    --from .openshell \
    --sync \
    -- bash

echo "Sandbox '$SANDBOX_NAME' ready."
echo "Connect with: openshell sandbox connect $SANDBOX_NAME"
```

Then the workflow is:

```bash
git clone git@github.com:you/my-project.git
cd my-project
./bootstrap.sh
# done — connect and go
openshell sandbox connect my-project
```

### One subtlety: nvim plugin installation

Your nvim config probably uses lazy.nvim or similar, which pulls plugins on first launch. Two options:

1. **Lazy approach** — just let it happen on first `nvim` invocation inside the sandbox. Requires github.com in your network allow list (which you'd want anyway).
2. **Hermetic approach** — add a `RUN` step in the Dockerfile that runs nvim headless to install plugins at image build time:

```dockerfile
# After copying nvim config
RUN su sandbox -c 'nvim --headless "+Lazy! sync" +qa 2>/dev/null || true'
```

Option 1 is fine for personal use. Option 2 is better if you want truly reproducible environments or if multiple people share the repo.

### Per-project customization

The beauty of checking `.openshell/` into each repo is that different projects can have different policies and different toolchains. A Rust project gets `rustup` in the Dockerfile; a Python ML project gets `conda`; the network policy for one might allow `huggingface.co`, another might not. The sandbox definition *is* part of the project, same as `.devcontainer/` or `docker-compose.yml`.

So yes — this is exactly the right pattern. One directory, one bootstrap command, hermetic and version-controlled.
