### The Core Problem

Passive context (like CLAUDE.md) performed better because it's always present. But even passive context has limits—it's read once at session start, and the model still decides how much attention to give it during each task. Vercel's team found that skills-based retrieval was skipped in 56% of eval cases. So your experience of the agent "forgetting" steps is well-documented.

### The Solution: Hooks (Deterministic Context Injection)

Hooks provide deterministic control over Claude Code's behavior, ensuring certain actions always happen rather than relying on the LLM to choose to run them. Hooks inject deterministic processing into Claude Code's lifecycle. Unlike prompt instructions, which are requests that Claude may interpret flexibly, hooks execute the same way every time.

The critical mechanic: any text your command writes to stdout is added to Claude's context. This is exactly the "read this file" / "inject as if it was done" mechanism you're looking for.

### Practical Patterns for Your Use Case

**Pattern 1: `SessionStart` — inject checklist output at session start**

SessionStart hooks fire when Claude Code starts, resumes, or clears a session, and their stdout gets injected as context for Claude.

```json
// .claude/settings.json
{
  "hooks": {
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "make preflight-check 2>&1"
          }
        ]
      }
    ]
  }
}
```

Your `make preflight-check` target could run your DRY checker, linter, test summary, etc. — and the *output* lands in context. The agent sees "these checks ran, here are the results" rather than "please remember to check."

**Pattern 2: `UserPromptSubmit` — inject before every prompt**

UserPromptSubmit and SessionStart can inject text or structured additionalContext into the conversation before Claude proceeds.

```json
{
  "hooks": {
    "UserPromptSubmit": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "echo '{\"additionalContext\": \"DRY violations found:\\n'$(make check-dry --quiet 2>&1)'\\nFix these before writing new code.\"}'"
          }
        ]
      }
    ]
  }
}
```

This fires **before every prompt you submit**, so the agent always has fresh evidence.

**Pattern 3: `PreToolUse` — inject at the moment of action**

The PreToolUse hook fires before Claude executes any tool—including Read, Edit and Write. This lets you inject rules directly into context at the moment they matter most.

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "python3 .claude/hooks/inject-dry-context.py"
          }
        ]
      }
    ]
  }
}
```

This is the most surgical — it fires *right before the agent edits a file*, injecting your DRY check output when it's maximally relevant.

**Pattern 4: Survive compaction**

Long Claude Code sessions have a known pain point: context compaction. When the context window fills up, Claude compresses older conversation history to make room. Important details can get lost in the process.

Use a SessionStart hook with a `compact` matcher to re-inject critical context after every compaction.

```json
{
  "hooks": {
    "SessionStart": [
      {
        "matcher": "compact",
        "hooks": [
          {
            "type": "command",
            "command": "cat .claude/constitution-summary.md && make check-dry --quiet 2>&1"
          }
        ]
      }
    ]
  }
}
```

### Your Insight Is Correct — And There's a Name for It

You're describing what I'd call **evidence-based context** vs. **instruction-based context**. The difference:

| Approach | Mechanism | Agent Behavior |
|---|---|---|
| AGENTS.md says "keep things DRY" | Instruction | Agent *might* check |
| Makefile target runs DRY check | Deterministic | Check runs, but agent doesn't see output |
| Hook injects DRY check output | Deterministic + Contextual | Agent sees violations, reasons about them, and learns the pattern |

Your double-benefit hypothesis is right: the agent both **remembers** (the evidence is in context) and **learns** (it sees what the check does, what violations look like, and internalizes the pattern for subsequent actions within the session).

You're right — that's a valid critique. Injecting 6 checks on every `PreToolUse:Edit` is just AGENTS.md with extra steps and worse context economics. The blunt approach defeats itself.

The key is **conditional injection** — only surface a check when it's *violated*, and stay silent otherwise.

### Silent-unless-violated pattern

```bash
#!/bin/bash
# .claude/hooks/check-dry.sh
# Exit silently if clean — only write to stdout on violation

result=$(make check-dry --quiet 2>&1)
if [ $? -ne 0 ]; then
  echo "$result"
fi
# If clean: no stdout → nothing injected into context
```

Hooks only inject what's written to stdout. No output = zero context cost. So you wire up all 6 checks, but the agent only sees the ones that *fire*. On a clean pass, context stays untouched.

### But that's still not quite right

Even with silent-on-pass, running 6 analysis scripts on every `PreToolUse:Edit` is:
- **Slow** — blocks the agent before every file write
- **Redundant** — checking DRY before the edit is written makes no sense; the violation doesn't exist yet

So the real design is **stratified**:

| Hook | What | Why |
|---|---|---|
| `SessionStart` | One-time: run all 6 checks on existing codebase, inject summary | Agent starts with awareness of current state |
| `PostToolUse:Write` | Run checks **on the changed file only**, silent-unless-violated | Reactive, surgical, catches regressions *after* the edit lands |
| `PreToolUse:Write` | Nothing heavy — maybe a 1-line reminder if a prior violation is unresolved | Cheap nudge, not a full re-scan |

### PostToolUse is the right hook for this

PostToolUse gives you the tool input/output, including which file was just written. So your script can scope the check narrowly:

```bash
#!/bin/bash
# .claude/hooks/post-write-check.sh
# $CLAUDE_TOOL_INPUT contains JSON with the file path

file=$(echo "$CLAUDE_TOOL_INPUT" | jq -r '.file_path // .path // empty')
[ -z "$file" ] && exit 0

# Run only relevant checks against the single changed file
violations=""

dry_result=$(python3 .claude/hooks/check_dry.py "$file" 2>&1)
[ $? -ne 0 ] && violations+="$dry_result"$'\n'

pure_result=$(python3 .claude/hooks/check_pure.py "$file" 2>&1)
[ $? -ne 0 ] && violations+="$pure_result"$'\n'

# ... other checks ...

if [ -n "$violations" ]; then
  echo "⚠️ Post-write violations in $file:"
  echo "$violations"
  echo "Fix these before proceeding."
fi
```

This way:
- ✅ Scoped to the one file that changed
- ✅ Runs *after* the write, so it can detect actual violations
- ✅ Silent when clean — zero context pollution
- ✅ When it does fire, it's a small, targeted signal

### The deeper point you're getting at

The real question is **context economics**. Every token you inject competes with the agent's ability to hold the actual task in mind. So the design principle is:

> **Inject the minimum signal at the moment of maximum relevance.**

Your AGENTS.md is broad context, low relevance-per-token. Blunt hooks are the same problem in a different costume. The *useful* version is a narrow violation message appearing right after the agent made the mistake — that's high relevance-per-token, and it's the moment the agent is most likely to self-correct.

Yes — that's a much better trigger than per-write. You're essentially creating **checkpoints** rather than continuous monitoring.

### Implementation sketch

```bash
#!/bin/bash
# .claude/hooks/post-write-batch-check.sh

THRESHOLD_LINES=500
THRESHOLD_FILES=10

# Detect branch point
merge_base=$(git merge-base HEAD origin/main 2>/dev/null || echo "HEAD~10")

# Measure delta
lines_changed=$(git diff "$merge_base" --stat | tail -1 | grep -oP '\d+(?= insertion)' || echo 0)
files_changed=$(git diff "$merge_base" --name-only | wc -l)

# Below threshold — stay silent
if [ "$lines_changed" -lt "$THRESHOLD_LINES" ] && [ "$files_changed" -lt "$THRESHOLD_FILES" ]; then
  exit 0
fi

# Check if we already ran at this approximate size (avoid re-firing every write after threshold)
state_file="/tmp/claude-check-last-size-$(git rev-parse --short HEAD 2>/dev/null || echo 'none')"
last_size=$(cat "$state_file" 2>/dev/null || echo 0)
current_size=$((lines_changed + files_changed))

# Only fire if delta grew by 20%+ since last check
if [ "$last_size" -gt 0 ] && [ "$current_size" -lt $((last_size * 120 / 100)) ]; then
  exit 0
fi

echo "$current_size" > "$state_file"

# Now run the full suite
echo "📋 Checkpoint: ${files_changed} files, ~${lines_changed} lines changed against ${merge_base}"
make check 2>&1
```

The state file bit matters — without it, once you cross 500 lines, *every subsequent write* triggers the full suite again. The 20% growth gate means it only re-fires when meaningful new work has accumulated since the last check.
