# improve <C-g>m so that it works to find artifacts of interest

After the start of ariadne datatypes, e.g. semi structured markdown files with rules, some processes are being managed by it, e.g. project and roadmap. if we compare those with issues, parley, there are similarities, they are just some semi structure markdown files as well, just the interpreter of the structure, whether it's by deterministic code or by agentic code. 

<C-g>m was created as a catch all to be able to find markdowns, as a viewing device. it can be improved along the following dimensions:

1/ maybe more depth. while default location of those datatypes are shallow e.g. data/project, data/roadmap, they might nest deeper. 
2/ to control noise, it's better to be able to find only datatype artifacts. this requires some directory or file naming pattern. for example, pensive always have yyyy-mm-dd-pensive-{slug}.md in the name. maybe we should follow that convention.
